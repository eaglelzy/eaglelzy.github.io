项目总结

掌途

  • viewpager轮播

    1. 在getCount是一个很大的值时(Integer.MAX_VALUE),setCurrentItem会ANR,因为里面有一个方法会对Count循环
      解决:不设置Integer.MAX_VALUE,设置数量为真实数量加上边上各两个缓冲长度.当滑动到头之后通过setCurrentItem重置位置.
    2. viewpager持有3个view,每次滑动都新创建一个view并remove边界的view,达不到复用目的,引入Pools.Pool将移除的view放入池中(池大小为4),新创建时从池里拿,拿不到在创建新的,减少频繁内存申请和释放
    3. 滑动不平滑,重置位置时没有平滑效果
      解决:通过监听viewpager的滑动事件 onPageScrollStateChanged(int state), 当state==0即停止滑动时,再调用setCurrentItem
      但是这样在快速滑动时,会出现ANR, 原因是因为快速滑动时,不会触发state==0的状态,可以通过在onPageSelected(int position)方法里面调用setCurrentItem.
      这样就不会出现ANR了,但是滑动会不平滑,可以通过加长缓冲长度降低概率

  • 多处UI类似时,统一定义成一个控件,差异用falg标志控制

  • 预加载,发表界面预加载行程,之前是wait,notify,后来改为回调队列
  • 爆炸效果,利用属性动画和动画插值器. 一个ValueAnimator一旦start后,可以调用getAnimatedValue获取时间流动比例,通过这个比例计算出每个碎片的位置,然后draw上.

掌途是一款基于高德sdk开发的景区导航的旅游产品.解决用户想去哪儿,怎么去和怎么玩的问题.
程序由3人开发.其中负责程序整体架构设计,性能优化以及图片上传,以及行程等功能模式的开发工作.
技术要点

  1. 自定义图片加载库(feature:主要借鉴glide的思路,精简了glide库.并增加了进度条功能.
    通过重复造轮子,深入理解了图片库设计中的一些瓶颈以及优化)
  2. 采用mvp架构,融合retrofit + rxjava,使程序结构层次更加清晰,单元测试更加方便
  3. 通过优化内存,图片处理,前后台切换等手段提升系统性能.
  4. 提高多渠道打包效率,由原先30分钟提高到现在2分钟,打包交给运营人员
  5. 解决方法数超过65536的各种坑
  6. 通过拆分为多进程,并将service设置为前台进程,实现进程保活方案
  7. 实现一些动画以及部分自定义控件(
    • 图片轮播:viewpager实现,解决anr以及平滑过度问题,采用Pools.Pool借鉴recyclerview思路封装一个adapter,实现view的复用,减少内存频繁申请和释放
    • 地图列表控件:通过flag配置来决定是否显示控件里面的元素
    • recyclerview分页功能:通过阅读源码找到grid布局和瀑布流布局时的分页方案)

技能清单:

  • 有扎实的java基础
  • 深入理解android
  • 善于利用工具提高效率(编写脚本,利用插件等)

    1. 系统缓存扫描是如何实现的?
      • 系统缓存是指所有已安装应用的data/data/packagename/cache和/sdcard/Android/data/packagename/cache文件夹
        根据系统自带SettingApp源码找到获取系统缓存的代码
        • 5.0以下是通过反射调用PackageManager.getPackageSizeInfo(String packageName, IPackageStatsObserver observer)实现的
          IPackageStatsObserver是Internal类,通过引入layoutlib.jar解决
      • 通过读取数据库clearpath5_other.db的syscache表,获取每个应用的提示语句,类型等
    2. sd卡上垃圾扫描如何实现
      • 每款应用在数据在sd卡都有对应缓存目录,通过对该目录扫描得到垃圾大小
    3. apk文件扫描
      • 使用复合扫描(Media扫描和sd卡扫描)
        1. media扫描
          cursor = cr.query(MediaStore.Files.getContentUri(“external”), null,

new String[]{"%.apk"}, null);
        2. sd卡扫描
          扫描指定目录下的.apk文件
    4. 日志和临时文件是以Media Stroe方式扫描的,同时加入服务器控制过滤
  • 提高垃圾扫描速度和准确率
    1. 扫描sd卡垃圾时, 对安装包进行排序,查询数据库将慢的app放到后面处理
    2. 扫描结果缓存10分钟

    1. 问题: 软件卸载界面不仅仅显示已安装的软件名称,还显示软件大小,以及所占用空间大小,使用频次,是否是高危app,还要进行病毒扫描, 任务很繁重. 怎么提升扫描用户体验?
      解决方案: 分为两个阶段: 
      1. 获取app列表, 对每个app进行频次,是否高危app进行判断, 每扫3个app刷新列表
2. 将病毒扫描和获取app大小, 以及获取文件残留信息, 等耗时扫描放到后台线程
    2. 如何实现静默卸载
      root的情况下才能实现静默卸载其原理为调用linux命令 pm uninstall packagename

    4. 备份功能怎么实现

    6. 耗电排行怎么实现
      每个厂商都有一个私有power_profile文件存储Android耗电信息
      耗电总量=CPU耗电+wake lock+数据流量+wifi耗电+传感器耗电

        events.getNextEvent(e);
        if (e.getEventType() == UsageEvents.Event.MOVE_TO_FOREGROUND) {
        pkgName = e.getPackageName();
clsName = e.getClassName();
        }
        }

基础知识

  1. 遇到的问题,怎么解决?

    • 渠道多,打包慢
      20多渠道打一次包要30多分钟,优化后只需打一个包1分钟左右,打包工作交给运营人员
      原理:apk本身也是一个压缩zip文件,通过解压apk向META-INF目录添加不同渠道
      的空文件,运行时从该目录读取渠道名称
    • 内存泄漏,OOM
      从一个页面进入另一个页面然后回到前一个界面,
      内存有明显升高,点击垃圾回收内存没有明星减少,说明有内存泄漏.
      单例持有Activity的Context

    • 65535坑

      1. 把启动页面专门放到一个进程中,在启动界面异步调用MultiDex.install(),这样主dex依赖集会尽可能小
        其他进程阻塞Application.attachBaseContext()调用,等启动页面加载完成后,再调用MultiDex.install()
        这样既不会anr,又能保证主dex最小依赖集

      2. 引入MultiDex后编译速度变慢,通过设置minsdk=21,提高编译效率

    • 代码质量是怎么保证的

    • UI效果
      recycleView加载更多的实现,通过阅读源码找到瀑布流分页加载更多的方案;
      swipeRefreshLayout下拉效果修改

图片优化

  • 组件思想

  • 申请到合适的内存
    判断设备是否为低内存设备(4.4或以上调用isLowRamDevice, 或版本小于3.0),如果为低内存设备,则/2
    最大可用内存设为总内存的0.4若是低内存设备则设置为0.33

    bitmap池大小

    array池大小
    MemoryCache大小
    默认array池大小为4M, MemoryCacheSize为2个屏幕内存大小,Bitmap池为4个屏幕内存大小,如果这两个之和大于可用内存,则按比例缩小.

  • ListView加载优化
    当前屏幕因为再次被访问机率不大故采用弱引用.
    采用两种缓存方式: active缓存,缓存当前屏幕的item,采用弱引用. 每当有新的item进入屏幕时,加入缓存,移除屏幕时,从缓存移除,同时加入到cache缓存.
    cache主要缓存屏幕外的item.active缓存为弱引用,是因为当前屏幕再次被访问的概率不大,缓存命中率应该低一些,这样cache缓存的就会尽可能多,提高缓存利用率

  • IdleHandler运用
    线程空闲时,能做一些事情,比如glide中在线程空闲时检测弱引用是否被回收,如果被回收,则从active缓存中移除对应的key

  • 图片生命周期处理
    每个页面都会添加一个无UI的Fragment,从而实现生命周期和OnTrimMemory的触发
    当触发到OnTrimMemory时,会根据不同的级别对内存进行回收处理

  • google的multidex原理
    解压/data/app/apkname.apk 得到class2.dex,class3.dex…classN.dex的zip列表.通过反射找到BaseDexClassLoader实例中的DexPathList类型的pathList变量.
    调用pathList的makeDexElements(ArrayList files, File optimizedDirectory)方法,将zip文件列表封装到Element数组中,
    修改pathList的dexElements数组变量的值,扩充上面的Element数组

    • multidex 坑
      1. 加载第二个dex时很慢
  • 池的复用
  • 使用Semaphore(信号量)配合线程池控制任务的提交速率,根据cpu数量设置Semaphore的上限,控制线程最大执行数量
  • RecycleView预加载
  • DiskLruCache 源码分析
  • DiskLruCacheWrapper put采用ReentrantLock(重入锁),避免使用synchronized，在多线程操作时提高的效率
  • GlideUrl
  • synchronized范围要尽可能的小
    LoadPathCache中AtomicReference的运用，提高并发效率

  • downsampler 源码分析
    利用inputStream的mark和reset复用流

    1. 图片decode优化,利用BitmapFactory.Options
      • 对图片进行合理缩放，计算inSampleSize值
      • 在图片为RGB_565格式时，使用抖动（设置options.inDither=true）会使图片看起来效果更好，尤其是渐变图片
      • 合理使用inBitmap复用内存，提高decode效率
      • KITKAT（19)之后可以利用Options.isScale,Options.inTargetDensity和Options.inDensity设置确切的比例，加载合适尺寸的图片

  • 数字使用下划线阅读起来会更清晰,比如:1000_000 10011111_10000000_11001010_11110000

  • 多进程访问SharedPreferences问题
